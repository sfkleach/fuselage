# Copilot Review

## Comment 1: src/archive.rs lines 148-156

The base64-alphabet heuristic is only applied to the first data line. If a later
line contains a non-base64 character (e.g., whitespace/indentation), this will
now be treated as a decode error rather than returning Ok(false) for “clearly
not base64”, which also diverges from the function’s doc comment. Consider
validating each non-comment data line (or each appended chunk) for allowed
characters before decoding, and returning Ok(false) when an invalid character is
encountered.

# Comment 2: src/archive.rs lines 157-161

Because the destination file is created and written incrementally, any
decode/write error after this point can leave a partially-written (corrupt)
output file at dest. To avoid leaving artifacts, write to a temporary file and
atomically rename on success, or ensure dest is removed/truncated on any error
path after creation.

# Comment 3: src/archive.rs lines 166-167

This only flushes a single chunk per iteration. If a single input line appends
more than B64_CHUNK bytes (or b64_buf grows by multiple chunks), the buffer can
grow large and defeat the streaming goal. Prefer a while b64_buf.len() >=
B64_CHUNK loop to decode all full chunks before continuing.

Suggested:
    Replace: |
        // Decode and flush whenever the buffer reaches a full chunk boundary.
        if b64_buf.len() >= B64_CHUNK {
    With: |
        // Decode and flush all complete chunks so large input lines do not
        // cause the buffer to retain multiple full chunks unnecessarily.
        while b64_buf.len() >= B64_CHUNK {

# Comment 4: src/archive.rs lines 174-176

b64_buf.drain(..B64_CHUNK) removes from the front of a String, which can require
shifting the remaining contents and becomes expensive if the buffer ever grows
large (e.g., unusually long lines). Consider using a Vec<u8> with a read offset
(or a ring buffer) to avoid repeated memmoves.

# Comment 5: src/archive.rs lines 185-192

There are tests for basic decoding, comments, and multiline inputs, but the new
streaming behavior isn’t covered for (1) invalid characters after the first data
line and (2) ensuring no partial output remains on mid-stream decode/write
failure. Adding tests for these cases would help lock in the intended Ok(false)
vs Err semantics and output cleanup behavior.
